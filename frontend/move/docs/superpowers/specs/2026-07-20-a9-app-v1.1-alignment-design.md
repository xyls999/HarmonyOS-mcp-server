# A9 App v1.1.0 接口与体验对齐设计

## 目标

将 HarmonyOS 手机 App 完整对齐 2026-07-19 发布的 A9 App 接口 v1.1.0。应用继续使用应用用户账号登录，只调用 `http://yuanzhe.tech/api/app/*`，不接触 D6 API Key，不连接 `/d6/*`，不使用 Mock 掩盖远端失败。

## 产品边界

- 保留四个主页面：首页、记录、助手、账号。
- 首页承载设备、传感器、区域状态和真实控制。
- 记录页承载操作日志、告警、安全事件和历史数据。
- 助手页承载文字输入、麦克风识别、AI 回复、图表和 TTS。
- 账号页承载用户资料、角色权限、登录日期和退出登录。
- 不恢复监测大页、主页场景快捷按钮、在线率、远程密钥管理、服务链接或内部配置入口。
- 场景 API、远程管理 API 等保留数据封装，不在主界面制造不必要入口。

## 网络与认证

- Base URL 固定为 `http://yuanzhe.tech`，业务路径统一为 `/api/app/*`。
- 登录、刷新、退出和当前用户接口严格使用文档字段。
- Access Token 提前五分钟自动刷新；401 刷新一次后重试，仍失败则退出登录。
- 对文档标记为需签名的 POST 请求生成 `X-App-Timestamp`、`X-App-Nonce` 和 `X-App-Signature`。
- 签名使用 SM3 摘要与 SM3-HMAC；若运行环境不支持则降级 SHA-256。
- 签名输入使用实际请求方法、完整路径和与发送内容完全一致的 JSON 字符串。
- 普通请求使用合理短超时；AI 对话读取超时不少于 120 秒。
- 400、403、404、426、504 映射为可执行的中文反馈，不把网关错误伪装成空数据或成功。

## 数据与控制

- 设备列表和详情使用服务端真实数据并归一化到现有 `Device` 模型。
- 开关请求使用 `{ "isOn": boolean }`。
- 参数控制使用文档要求的扁平请求体，例如 `action`、`temperature`、`mode`、`fan_speed`，不再嵌套 `params`。
- 门禁打开前弹出密码输入；关闭门禁无需密码。
- 传感器列表和历史按 `sensorId`、`hours` 查询。
- 记录页分别加载 operations、alerts、security/events，单个接口失败不阻断其他分区。
- 504 明确显示 A9 网关无响应；无数据与加载失败使用不同状态。

## AI 对话

- `/api/app/chat/send` 发送用户原文，不在客户端推断设备动作。
- 支持 `reply` 文本、`chartData`、`tts_audio` 和 `voiceSequence`。
- 语音输入读取手机麦克风并将识别结果填入输入框，由用户发送。
- AI 请求期间展示完整但克制的加载动效，完成后再替换为回复。
- 服务端音频可用时优先保留音频地址；本地 TTS 作为可用的朗读能力。
- 401、403、504 和超时回复作为错误状态显示，不伪造成 AI 正常回答。

## 视觉与交互

- 借鉴 `Harmon_LandscapeControl_OpenHarmony` 的淡蓝/黑配色、细线框、块面层级、艺术字标识和明确选中态。
- 针对 Nova 12 竖屏重新排版，不照搬 RK3568 横屏尺寸。
- 控制装饰数量，避免虚线、杂乱短线、重复小字和主题化命名。
- 页面切换保留连续动画，但不遮挡内容或延迟真实操作。
- 所有按钮具备禁用、执行中、成功或失败反馈；不使用颜色作为唯一状态提示。

## 验证

- 使用真实应用用户登录公网服务。
- 验证 `/api/app/auth/me`、devices、sensors、operations、alerts 和 chat/send。
- 对真实设备仅执行用户明确允许的低风险读取；控制接口使用不存在设备或不改变状态的方式验证转发。
- 完整运行 Hvigor `assembleHap`。
- 识别 HDC 目标型号，只安装到 Nova 12，不部署 RK3568。
- 启动应用并检查关键 hilog，不把系统能力警告误报为构建失败。
