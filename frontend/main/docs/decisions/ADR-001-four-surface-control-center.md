# ADR-001: 四栏 OpenHarmony 主控界面

## 状态

Accepted

## 日期

2026-07-18

## 背景

主控 App 既需要自然语言助手，也必须提供无需 AI 的真实设备按钮、实时数据管理、联动设置和主人评分入口。设备为横屏主控，原两栏界面无法承载这些功能，启动窗口还存在黑色边缘和系统栏覆盖。

## 决策

- 底部固定为“助手 / 设备 / 数据管理 / 设置”四栏。
- 设备栏直接调用 `DeviceApi`，提供灯、风扇、空调、窗帘、报警和门禁的细分按钮；门禁密码仅保留在当前输入状态并随单次请求发送。
- 数据栏继续 10 秒刷新日志、设备和环境，并接入服务端确定性 `/api/log/chart`。
- 设置栏控制联动自动管理、主动 AI、毫米波存在感知，展示多协议状态和 QQ 配置，并提交 0–10 评分与改进建议。
- App 行为通过脱敏遥测进入后端上下文；密码、Token、密钥永不进入遥测。
- Ability 强制横屏，启动背景与页面主题一致，并移除加载侧边轨道。

## 备选方案

### 所有控制都交给助手

拒绝。现场主控需要可发现、可重复、低延迟的按钮控制，不能把基础操作依赖于自然语言解析和上游模型可用性。

### 继续使用两栏并将设置放入弹窗

拒绝。联动、反馈、协议和安全状态属于长期使用的一级信息，弹窗会降低可发现性并造成横屏布局浪费。

## 后果

- 新页面依赖后端 v6 新增的警戒、遥测、协议和日志图表接口。
- UI 使用真实接口失败状态，不再把请求失败伪装为 Mock 成功。
- OpenHarmony 本地发布签名必须使用 `tools/build-openharmony.ps1`，不能直接安装 DevEco 调试签名覆盖 release provision 应用。
