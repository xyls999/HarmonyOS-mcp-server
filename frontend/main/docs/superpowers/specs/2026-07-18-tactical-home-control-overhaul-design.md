# 明日家居战术主控重构设计

## 1. 目标与范围

在 D6 竖屏主控设备上重构“助手、设备、数据管理、设置”四个页面。保留“明日家居”Logo、现有明暗配色关系、真实设备协议、门禁密码约束和后端部署路径，不改变底部四栏信息架构。

本轮同时修复四类真实故障：

1. 厨房联动总开关关闭后仍生成被动警戒、恢复动作或语音汇报。
2. 厨房烟雾和热敏信号抖动造成重复告警与重复恢复。
3. 设置页开关点击后视觉不变，计划确认配置无法持久化。
4. 页面切换底部进度条不响应，动画中断后可能残留遮罩。

## 2. 设计判断

- 模式：保留品牌和信息架构的整体重构。
- 受众：固定在家中的竖屏触控主控设备使用者。
- 视觉语言：明日方舟式工业战术终端，不复刻游戏资产。
- DESIGN_VARIANCE：8。使用偏轴构图、切角和强弱分区建立辨识度。
- MOTION_INTENSITY：7。包含有意义的页面过渡、主题切换和按压反馈，但只动画位移、缩放、旋转与透明度。
- VISUAL_DENSITY：5。保留足够设备信息，不使用小字和驾驶舱式数据堆叠。

## 3. 竞品原则

- 借鉴 Apple Home 的直接控制原则：点击设备主控区快速切换，精细参数在同一设备区域展开，避免把所有操作同时铺满。
- 借鉴 Home Assistant Sections 的分区原则：按真实用途组织设备，分区高度由内容决定，不保留空卡或假设备。
- 超越点：所有按钮拥有立即状态反馈、明确的执行中和失败回滚；AI 计划展示、确认与执行使用同一份不可变命令摘要；系统告警拥有可解释证据和生命周期去重。

## 4. 视觉系统

### 4.1 保留项

- 保留明日家居 Logo、绿色主强调色、现有明暗主题关系和四栏底部导航。
- 保留竖屏方向。
- 保留系统 SymbolGlyph 图标族，不引入第三方图标或图片依赖。

### 4.2 新视觉规则

- 页面使用冷白或近黑底色，面板只使用同主题内的轻微层级差。
- 唯一强调色继续使用现有绿色；危险状态使用现有红色语义色，不新增紫色、蓝紫渐变或霓虹外发光。
- 主要容器采用 0-4vp 小圆角和单侧切角；圆形只用于语义明确的状态或图标按钮。
- 不使用背景网格、装饰性十字线、版本号、序号眉题、伪终端小字或无意义英文。
- 可见正文最小 14vp，关键按钮文字 15-17vp，页面标题 26-30vp。
- 控件视觉可紧凑，但触控区域至少 44vp。

### 4.3 页面结构

#### 助手

- 对话流仍是主内容，普通用户操作只进入上下文，不作为助手消息显示。
- 自动汇总仅在有变化、真实风险或具体建议时进入对话。
- 状态汇报不要求评分；只有成功改变至少一个真实设备状态的事件显示评分。
- 计划卡显示原因、设备、动作和参数，并提供“执行计划”和“取消”两个明确按钮。
- 图表只使用真实数据；无数据时显示简洁空状态，不制造 1 或固定倍率数据。

#### 设备

- 分类为门禁与出入口、温控与空气、照明与遮阳、其他设备。
- 删除旧 mock 设备 `exhaust_01` 抽风机和 `fan_01` 客厅吊扇；保留真实 `fan_02` 卫生间换气扇。
- 普通设备使用紧凑双态开关；空调使用模式按钮、温度加减和应用操作；门禁每次要求密码。
- 烟雾、热敏、温湿度不在设备页展示。

#### 数据管理

- 日志、设备历史和环境数据继续实时更新。
- 烟雾和热敏只出现在日志与真实报警记录中，不展示为设备控制卡。
- 日志总数来自后端真实总数，列表增量刷新，不因定时刷新跳回顶部。

#### 设置

- 保留联动自动管理、主动 AI 警戒、评分自动触发、计划确认后执行、毫米波存在感知、清空助手和退出应用。
- 行为评分不在设置页展示。
- 每个开关为独立响应式组件，点击先更新本地状态，接口失败才回滚。
- 加载接口不覆盖正在提交的本地状态；成功后以服务端返回值校准。

## 5. 动效系统

### 5.1 页面切换

- 使用一个过渡时长常量派生进入、页面交换、100% 保持和退出节点，不再写多组互相错位的定时值。
- 底部进度轨改为直接读取 `@State` 的自绘轨道和移动方块，不通过普通 Builder 参数传递状态。
- 进度必须从 0 连续到 100，100% 至少保持一帧。
- 页面切到后台或组件销毁时，取消计时器并原子复位遮罩和切换锁。
- 重复点击时允许最新导航意图接管，不永久拒绝后续切换。

### 5.2 明昼切换

- 明昼台词：`天光接管，居所进入明昼模式`。
- 暗夜台词：`夜幕落下，系统转入低光守护`。
- 台词在主题过渡遮罩中只显示一行，同时通过现有 TTS 队列播报一次。
- 主题切换使用遮罩扫描、Logo 切片位移和明暗面交替，不使用背景网格或大型旋转圆环。
- 所有自动动效可被系统减少动态效果设置降级为短淡入。

### 5.3 微交互

- 按下在 100ms 内给出缩放或位移反馈。
- 异步按钮具有执行中锁定、成功确认和失败回滚。
- 页面内容进入采用 30-50ms 的短错峰，最多作用于首屏关键块，不对长列表逐项持续动画。

## 6. 厨房报警状态机

### 6.1 总开关静默门

当“联动自动管理”关闭时：

- 保留原始传感器采集和日志持久化。
- 不创建 guard incident。
- 不进入助手 feed。
- 不执行蜂鸣器或换气扇动作，包括恢复动作。
- 不发送 TTS、QQ 或其他通知。
- 不生成待评分事件。

`kitchenAlarm.enabled` 关闭时使用相同静默规则，仅作用于厨房报警。

### 6.2 去误报与生命周期

- 启动后 20 秒为采样稳定期，只记录原始数据，不触发厨房告警。
- 厨房轮询每秒一次，连续 3 次报警状态一致才进入报警态。
- 连续 5 次清除状态一致才进入恢复态，避免 0/1 抖动反复开关设备。
- 同一报警生命周期只创建一个 incident；重复轮询和 UDP 数据更新证据与最后时间，不重复播报。
- 已恢复 incident 在冷却期内再次短暂抖动只更新证据，不新建事件。
- 热敏原始毫伏变化只用于日志和趋势，不直接等同于 `isAlert`。
- 只有明确的 `smoke_alarm`、`temp_alarm` 或综合 `alarm` 确认态参与报警判定。

## 7. 设置和计划执行数据流

### 7.1 设置

1. 用户点击开关。
2. 前端为对应开关建立独立提交状态并立即改变视觉位置。
3. 前端 POST 完整配置。
4. 后端白名单接受并持久化 `planConfirmation`。
5. 后端返回完整规范化配置。
6. 前端使用返回值校准；失败时只回滚本次开关。

定时刷新在开关提交期间不得覆盖本地乐观状态。

### 7.2 计划确认

- 待确认计划按用户、客户端会话和对话 ID 隔离，并设置 5 分钟过期时间。
- 模糊意图、否定句、条件句和没有明确设备目标的表达不得直接执行。
- 配置变更不能混在设备控制计划中；必须单独展示旧值和新值并确认。
- 计划卡展示的命令列表与后端保存的命令列表完全一致，并绑定摘要和 nonce。
- 关闭“计划确认后执行”只允许明确肯定的设备命令直达；模糊命令仍然只生成建议，不自动猜测执行。

## 8. 错误处理与安全边界

- 设备接口失败时保留原状态并显示明确失败反馈。
- 门禁密码不写入日志、上下文、遥测或计划卡。
- 不删除历史反馈或告警数据；新逻辑只阻止错误新增。
- 不改变下位机协议，不修改 IP、端口或真实设备 ID。
- 自定义设备仍可通过现有注册接口注入，但两个已确认不存在的旧 mock ID 永久过滤。

## 9. 测试与验收

### 9.1 后端测试

- 总开关关闭时，报警和恢复均不产生 incident、动作、TTS、QQ 或助手 feed。
- 连续 2 次报警不触发，第 3 次触发一次；报警持续期间不重复触发。
- 连续 4 次恢复不关闭，第 5 次只恢复一次。
- 热敏毫伏漂移不生成风险事件。
- `planConfirmation` 更新后重建 AdaptiveGuard 仍保持状态。
- 失败动作和未改变状态的动作不开放评分。
- `/api/devices` 不返回 `fan_01` 或 `exhaust_01`，仍返回 `fan_02`。

### 9.2 前端契约与构建

- 所有设置开关直接绑定独立状态，不通过普通 Builder 字符串参数间接读取。
- 计划开关点击状态、服务端返回和重进页面保持一致。
- 进度轨直接读取响应式状态，过渡结束原子复位。
- 可见字号不低于 14vp，主要触控区不低于 44vp。
- 页面不存在背景网格、虚假设备、小字号元数据和设置页评分面板。
- Python 契约测试全部通过，ArkTS 构建成功。

### 9.3 D6 真机验收

- 安装并启动最新签名 HAP。
- 逐个点击五个设置开关，检查立即变化、接口返回、重进页面和后端重启后的持久化。
- 在联动关闭状态向警戒模块注入测试报警快照，确认无播报、无动作和无助手事件。
- 捕获页面切换 0.3s、0.8s、1.4s 和完成帧，确认进度连续增长且不残留遮罩。
- 检查明昼台词、主题色、助手计划卡、设备分类、数据日志和设置退出按钮。
- 检查 hilog 与后端日志，无 ArkUI 异常、线程崩溃或重复告警。
